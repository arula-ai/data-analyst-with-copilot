# Glossary — Data Analysis Lab Reference

---

## Lab Framework Terms

**RIFCC-DA**
The prompt framework used in this lab. An adaptation of the RIFCC framework for data analysis work. The five fields — Role, Inputs, Format, Constraints, Checks — constrain Copilot's responses to produce auditable, policy-compliant analytical output. The "DA" suffix indicates the data analysis variant, which adds financial data handling rules to the base framework.

**Agent Mode**
A preset role and behavioral ruleset you activate in Copilot Chat to constrain how it responds. In this lab, each of the six stages has a corresponding agent mode (e.g., "Data Profiling Analyst," "Data Cleaning Engineer"). Activating a mode tells Copilot which expert mindset to adopt, what rules to follow, and what output format to produce.

**Prompt Contract**
A written RIFCC-DA prompt you commit to before starting a stage. Calling it a "contract" reinforces that you are specifying exactly what you expect from Copilot — and that you are responsible for reviewing what it delivers.

**Hand-off Checkpoint**
A summary prompt at the end of each stage that asks Copilot to capture what you accomplished in 3 bullet points. This creates a record of what each stage produced and helps you synthesize findings before moving to the next stage.

## Dataset Terms

**Alert / Fraud Alert**
An automated notification generated by the transaction risk system when a transaction pattern deviates from established risk thresholds. Each row in `transaction_alerts.csv` represents one alert case.

**Attrition / Fraud Confirmation**
In this dataset, `fraud_confirmed` indicates whether an alert was ultimately confirmed as genuine fraud (1) or not (0). Confirmation requires analyst review within 30 days of alert creation.

**Risk Score**
A normalized score from 0.0 to 1.0 output by the alert engine that reflects the strength of the fraud signal. 0.0 = no detectable risk signal; 1.0 = maximum risk signal. Values above 1.0 are data entry errors.

**Analyst Confidence**
A 0–10 self-rating provided by the reviewing analyst indicating their confidence that an alert represents genuine fraud. 9–10 = high confidence, 5–8 = moderate, 0–4 = low. The value -1 is a legacy system code meaning "not rated" and is not valid on the 0–10 scale.

**Sentinel Value**
A special numeric value used in a system to indicate "no data," an error condition, or a missing response — coded as a real number rather than null. In this dataset: `prior_alerts_90d = 999` and `analyst_confidence = -1` are sentinel values. Including sentinels in calculations (means, counts, rates) produces misleading results.

**Alert Type**
The category of fraud signal that triggered the alert. Valid types: Velocity Check, Large Transaction, Geographic Anomaly, Account Takeover, Pattern Match.

**Client Segment**
The classification of the account holder at the time of the alert: Retail, Business, Premier, or Institutional. Fraud patterns often vary significantly across segments.

## Data Analysis Terms

**Data Profiling**
The process of examining a dataset to understand its structure, content, quality, and statistical properties — before any analysis or transformation begins. Profiling answers: how many rows? how many nulls? what values appear? what ranges are present? are there schema violations?

**Data Cleaning / Transformation**
Modifying data to fix errors, inconsistencies, and missing values to prepare it for analysis. In regulated financial environments, every cleaning decision requires a written justification. "The model needs clean data" is not a justification.

**EDA (Exploratory Data Analysis)**
Open-ended analysis conducted to discover patterns, anomalies, and relationships in a dataset — usually before building a formal model. In this lab, EDA is always framed around a specific business question to avoid unfocused exploration.

**Imputation**
Replacing missing values with a calculated or assumed substitute (e.g., mean, median, or a business-rule-based value). Every imputation in this lab must be documented with the assumption it makes. Imputing without documentation is a policy violation.

**Deduplication**
Removing duplicate records from a dataset. In this dataset, 12 `alert_id` values appear more than once. Deduplication must specify which duplicate to keep (e.g., first occurrence, most recent) — not just drop arbitrarily.

**Outlier**
A data point significantly different from others in the same column. May indicate a data entry error (e.g., `risk_score = 1.35`) or a genuine extreme case (e.g., a very large transaction). Outliers must be investigated before being dropped.

**Correlation**
A statistical measure of how two variables move together. Does not imply causation. In fraud analysis: a high correlation between `prior_alerts_90d` and `fraud_confirmed` is interesting, but does not mean prior alerts cause fraud.

**False Positive**
An alert flagged by the system that turns out not to be fraud upon investigation. A high false positive rate reduces analyst efficiency and increases operational cost.

**Confirmation Bias**
The tendency to interpret data in a way that confirms a pre-existing belief. In fraud analysis: assuming all high-risk-score alerts are fraud before reviewing the evidence. A key reason why explicit assumptions and limitations are required at every stage.

## Technical Terms

**Pandas DataFrame**
A table-like data structure in Python used for data manipulation. Think of it as an in-memory spreadsheet that supports filtering, grouping, transformation, and statistical analysis through code.

**Matplotlib / Seaborn**
Python libraries for creating charts and visualizations. Matplotlib provides low-level control; Seaborn provides higher-level, statistically oriented chart types built on top of Matplotlib. Both are approved for this lab.

**Python Script**
A standalone script that executes code and exports charts to HTML format. Stages 3 and 4 use your generated `.py` scripts as the working environment.

**CSV (Comma-Separated Values)**
A plain text file format for tabular data. Each row is a record; fields are separated by commas. `transaction_alerts.csv` is this format — it can be read by pandas, Excel, and most data tools.

**Schema**
A documented description of a dataset's structure — column names, data types, valid ranges, and business rules. `data/schema.md` is the schema for this lab. When Copilot's output conflicts with the schema, the schema wins.

## Responsible Use Terms

**PII (Personally Identifiable Information)**
Data that can identify a specific individual — directly (name, SSN, account number) or indirectly (masked account combined with region and transaction amount). The `account_masked` field is PII-adjacent: it doesn't directly identify someone, but it could in combination with other fields.

**Data Classification**
A system of tiers categorizing data by sensitivity: Public, Internal, Confidential, Restricted. Classification determines who can access data, what tools can process it, and what approval is required before AI-assisted analysis. See `reference/responsible_use.md` for Hartwell's tier definitions.

**Responsible Use**
Operating AI tools within organizational policies that protect security, privacy, data integrity, and regulatory compliance. In this lab: not sending `account_masked` to Copilot in raw form, documenting all assumptions, and reviewing every generated script before running it.

**Code Review**
The process of reading generated code to verify correctness, safety, policy compliance, and alignment with business rules. In this lab, Stage 5 is the formal code review stage. In practice, every stage requires informal code review before execution.

**SAR (Suspicious Activity Report)**
A formal report filed with financial regulators (FinCEN in the US) when a financial institution suspects money laundering, fraud, or other financial crimes. Not directly referenced in this dataset, but the context for why fraud confirmation matters — confirmed fraud cases may trigger SAR filings.
