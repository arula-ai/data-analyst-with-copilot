# Glossary — Data Analysis Lab Reference

---

## Lab Framework Terms

**RIFCC-DA**
The prompt framework used in this lab. An adaptation of the RIFCC framework for data analysis work. The five fields — Role, Inputs, Format, Constraints, Checks — constrain Copilot's responses to produce auditable, policy-compliant analytical output. The "DA" suffix indicates the data analysis variant, which adds financial data handling rules to the base framework.

**Agent Mode**
A preset role and behavioral ruleset you activate in Copilot Chat to constrain how it responds. The lab features four active agent modes: Data Profiling Analyst, Data Cleaning Engineer, Exploratory Data Analyst, and Visualization Architect. Each mode corresponds to a phase or focus area within a scenario. Activating a mode tells Copilot which expert mindset to adopt, what rules to follow, and what output format to produce.

**Prompt Contract**
A written RIFCC-DA prompt you commit to before starting a phase. Calling it a "contract" reinforces that you are specifying exactly what you expect from Copilot — and that you are responsible for reviewing what it delivers.

**Hand-off Checkpoint**
A summary prompt at the end of each phase that asks Copilot to capture what you accomplished in 3 bullet points. This creates a record of what each phase produced and helps you synthesize findings before moving to the next phase.

## Dataset Terms

**Scenario**
The lab is structured into three independent scenarios, each within Arula Financial Group but with distinct business contexts, datasets, and analytical focuses: Scenario A (Treasury), Scenario B (RCA — Root Cause Analysis), and Scenario C (Modernization).

**Scenario A: Treasury (Meridian Asset Management)**
Focuses on treasury payment monitoring. Data: `treasury_payments.xlsx` (500 rows, 14 columns). PII field: `counterparty_masked`. Key columns include alert classifications, risk scores, and anomaly indicators. Sentinel values: `prior_alerts_90d = 999`, `analyst_confidence = -1`. Alert types: Velocity Check, Large Transaction, Geographic Anomaly, Account Takeover, Pattern Match.

**Scenario B: RCA (Orion Payments Inc.)**
Focuses on payment service logs and root cause analysis. Data: `rca_app_logs.csv` (300 rows, 9 columns). PII field: `user_id_masked`. Key columns: timestamp, service_name, environment, log_level, error_code, message, request_id, response_time_ms, user_id_masked. Supports investigation of system failures and performance issues across 5 microservices.

**Scenario C: Modernization (Centrix Financial Systems)**
Focuses on mainframe feature evaluation for system modernization. Data: `mainframe_usage.xlsx` (400 rows, 10 columns). Key columns: feature_id, feature_name, team, monthly_active_users, avg_response_ms, error_rate_pct, last_accessed_date, legacy_flag, modernization_priority, estimated_migration_effort_days. Sentinel value: `estimated_migration_effort_days = 9999` (effort not yet assessed). Used to prioritize which legacy features to migrate or retire.

**Alert / Anomaly Indicator**
(Scenario A) An automated notification generated by the treasury risk system when a transaction pattern or payment behavior deviates from established risk thresholds. The `anomaly_confirmed` column indicates whether an alert was substantiated upon analyst review.

**Analyst Confidence**
A 0–10 self-rating provided by the reviewing analyst indicating their confidence in a flagged anomaly or risk. 9–10 = high confidence, 5–8 = moderate, 0–4 = low. The value -1 is a legacy system code meaning "not rated" and is not valid on the 0–10 scale.

**Sentinel Value**
A special numeric value used in a system to indicate "no data," an error condition, or a missing response — coded as a real number rather than null. Examples: `prior_alerts_90d = 999` (Scenario A), `estimated_migration_effort_days = 9999` (Scenario C). Including sentinels in calculations (means, counts, rates) produces misleading results.

**Alert Type**
(Scenario A) The category of transaction signal that triggered an alert. Valid types: Velocity Check, Large Transaction, Geographic Anomaly, Account Takeover, Pattern Match.

## Data Analysis Terms

**Data Profiling**
The process of examining a dataset to understand its structure, content, quality, and statistical properties — before any analysis or transformation begins. Profiling answers: how many rows? how many nulls? what values appear? what ranges are present? are there schema violations?

**Data Cleaning / Transformation**
Modifying data to fix errors, inconsistencies, and missing values to prepare it for analysis. In regulated financial environments, every cleaning decision requires a written justification. "The model needs clean data" is not a justification.

**EDA (Exploratory Data Analysis)**
Open-ended analysis conducted to discover patterns, anomalies, and relationships in a dataset — usually before building a formal model. In this lab, EDA is always framed around a specific business question to avoid unfocused exploration.

**Imputation**
Replacing missing values with a calculated or assumed substitute (e.g., mean, median, or a business-rule-based value). Every imputation in this lab must be documented with the assumption it makes. Imputing without documentation is a policy violation.

**Deduplication**
Removing duplicate records from a dataset. In this dataset, 12 `alert_id` values appear more than once. Deduplication must specify which duplicate to keep (e.g., first occurrence, most recent) — not just drop arbitrarily.

**Outlier**
A data point significantly different from others in the same column. May indicate a data entry error (e.g., `risk_score = 1.35`) or a genuine extreme case (e.g., a very large transaction). Outliers must be investigated before being dropped.

**Correlation**
A statistical measure of how two variables move together. Does not imply causation. In anomaly analysis: a high correlation between `prior_alerts_90d` and `anomaly_confirmed` is interesting, but does not mean prior alerts cause anomalies.

**False Positive**
An alert flagged by the system that turns out not to be fraud upon investigation. A high false positive rate reduces analyst efficiency and increases operational cost.

**Confirmation Bias**
The tendency to interpret data in a way that confirms a pre-existing belief. In fraud analysis: assuming all high-risk-score alerts are fraud before reviewing the evidence. A key reason why explicit assumptions and limitations are required at every stage.

## Technical Terms

**Pandas DataFrame**
A table-like data structure in Python used for data manipulation. Think of it as an in-memory spreadsheet that supports filtering, grouping, transformation, and statistical analysis through code.

**Plotly Express**
A Python library for creating interactive, publication-quality charts and dashboards. Charts are exported as standalone HTML files and support hover tooltips, filtering, and responsive design. Plotly Express is the approved visualization library for this lab; it replaces matplotlib and seaborn.

**Python Script**
A standalone script that executes code and exports charts to HTML format. Phase 3 (Visualize) uses your generated `.py` scripts as the working environment.

**Data Formats**
The lab uses two tabular data formats: CSV (Comma-Separated Values) — plain text, rows as records, fields separated by commas (e.g., `rca_app_logs.csv`) — and XLSX (Microsoft Excel binary format) — preserves formatting and multiple sheets (e.g., `treasury_payments.xlsx`, `mainframe_usage.xlsx`). Both are read by pandas and openpyxl.

**Schema**
A documented description of a dataset's structure — column names, data types, valid ranges, and business rules. Schema files are located at `data/treasury_schema.md`, `data/rca_schema.md`, and `data/mainframe_schema.md` for the three scenarios. When Copilot's output conflicts with the schema, the schema wins.

## Responsible Use Terms

**PII (Personally Identifiable Information)**
Data that can identify a specific individual — directly (name, SSN, account number) or indirectly (masked account combined with region and transaction amount). In this lab, PII-adjacent fields include: `counterparty_masked` (Scenario A) and `user_id_masked` (Scenario B). These masked identifiers don't directly identify someone, but they could in combination with other fields.

**Data Classification**
A system of tiers categorizing data by sensitivity: Public, Internal, Confidential, Restricted. Classification determines who can access data, what tools can process it, and what approval is required before AI-assisted analysis. See `reference/responsible_use.md` for Arula Financial Group's tier definitions.

**Responsible Use**
Operating AI tools within organizational policies that protect security, privacy, data integrity, and regulatory compliance. In this lab: not sending masked identifiers or raw transactional context to Copilot in potentially de-anonymizing combinations, documenting all assumptions, and reviewing every generated script before running it.

**Code Review**
The process of reading generated code to verify correctness, safety, policy compliance, and alignment with business rules. In practice, every phase requires code review before execution to ensure data safety and analytical integrity.

**SAR (Suspicious Activity Report)**
A formal report filed with financial regulators (FinCEN in the US) when a financial institution suspects money laundering, fraud, or other financial crimes. (Scenario A) Confirmed anomalies may trigger SAR filings, making rigorous data analysis and alert confirmation critical.
